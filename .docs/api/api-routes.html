Les routes marquées "Auth optionnelle" acceptent uniquement le cookie community rs_app_session mais fonctionnent aussi sans session. Le cookie staff rs_admin_session n'est jamais interprété par les routes applicatives.

Conventions

Erreur standard

Toutes les erreurs HTTP passent par le middleware central.

{
  "error": {
    "message": "Route not found",
    "code": "ROUTE_NOT_FOUND"
  }
}

Pagination standard

page et limit sont des entiers positifs. La limite générale est plafonnée à 50.

{
  "items": [],
  "pagination": {
    "page": 1,
    "limit": 12,
    "totalItems": 0,
    "totalPages": 0,
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}

Auth

Les sessions community et staff utilisent des cookies, audiences JWT, durées et tables SQL distincts. Le login community pose rs_app_session pour 7 jours par défaut. Le login staff vérifie le mot de passe puis une assertion WebAuthn avec vérification utilisateur avant de poser rs_admin_session pour 8 heures par défaut. Les routes sensibles sont limitées par le rate limiter auth.

Méthode Route Auth Body / Query Réponse
POST /api/v1/auth/register Public mail, username, password. Password min 8 caractères, username min 3. Le serveur attribue accountType: "community". 201 { "user": ..., "message": "..." }.
POST /api/v1/auth/login Public mail, password. 200 { "user": ... } + Set-Cookie: rs_app_session=.... Toute identité staff est refusée avec des credentials invalides.
GET /api/v1/auth/me Session community Aucun. 200 { "auth": ... }.
POST /api/v1/auth/logout Public Cookie community optionnel. Révoque et supprime uniquement rs_app_session. 200 { "ok": true }.
POST /api/v1/admin/auth/mfa/enrollment/options Public invitationToken. 200 { "flowId": "...", "publicKey": ... }. Options de création WebAuthn avec vérification utilisateur obligatoire.
POST /api/v1/admin/auth/mfa/enrollment/verify Public flowId, invitationToken, password, credential WebAuthn. 200 { "userId": 1, "mfaEnrolled": true }. Active le staff atomiquement, sans créer de session.
POST /api/v1/admin/auth/login/options Public mail, password. 200 { "flowId": "...", "publicKey": ... }, sans cookie. Erreurs : STAFF_INVITED, STAFF_LOCKED, STAFF_DISABLED, STAFF_MFA_REQUIRED.
POST /api/v1/admin/auth/login/verify Public flowId, credential WebAuthn. 200 { "user": ... } + Set-Cookie: rs_admin_session=.... Erreur : AUTH_INVALID_MFA_ASSERTION.
GET /api/v1/admin/auth/me Session staff + MFA Aucun. 200 { "auth": ... }.
POST /api/v1/admin/auth/logout Public Cookie staff optionnel. Révoque et supprime uniquement rs_admin_session. 200 { "ok": true }.
POST /api/v1/auth/validate-email Public token. 200 { "ok": true, "message": "...", "user": ... }.
POST /api/v1/auth/resend-validation-email Public mail. 200 message générique.
POST /api/v1/auth/forgot-password Public mail. 200 message générique.
POST /api/v1/auth/reset-password Public token, password. Alias accepté: newPassword. 200 confirmation.

Recipes

Lecture publique des recettes publiées. La création, la mise à jour, la soumission et l’archivage personnels exigent rs_app_session pour un compte community actif ; un cookie staff reçoit 401 AUTH_NO_TOKEN.

Méthode Route Auth Body / Query Réponse
GET /api/v1/recipes Auth optionnelle page, limit, q, categoryId, tagIds, excludedTagIds, ingredientIds, excludedIngredientIds, maxTotalTimeMinutes. 200 résultat paginé des recettes publiées.
GET /api/v1/recipes/search Auth optionnelle Mêmes filtres que /api/v1/recipes. Les quatre listes d'identifiants sont séparées par des virgules et dédupliquées. Une recette doit posséder tous les tagIds et ingredientIds, et elle est écartée dès qu'elle possède un excludedTagIds ou excludedIngredientIds. Un identifiant ne peut pas être à la fois inclus et exclu dans une même famille. 200 résultat paginé.
GET /api/v1/recipes/recent Auth optionnelle limit optionnel, défaut 12, maximum 20. 200 liste des recettes récentes.
GET /api/v1/recipes/:slug Auth optionnelle slug requis. 200 détail recette, ou 404 NOT_FOUND.
GET /api/v1/recipes/me Session page, limit. Défaut limit 10, maximum 50. 200 recettes de l’utilisateur.
POST /api/v1/recipes Community active Body recette. title requis, min 5 caractères. 201 brouillon créé.
GET /api/v1/recipes/me/:id Session id entier positif. 200 détail de la recette de l’utilisateur.
PATCH /api/v1/recipes/me/:id Community active Body recette. title requis, min 5 caractères. 200 brouillon mis à jour.
POST /api/v1/recipes/me/:id/submit Community active id entier positif. 200 résultat de soumission.
POST /api/v1/recipes/me/:id/archive Community active id entier positif. 200 { "ok": true }.

Admin users

Administration des utilisateurs bannis, profil admin et journal de modération. La consultation exige users.read et le bannissement ou la réactivation users.moderate.

Méthode Route Auth Body / Query Réponse
GET /api/v1/admin/users/banned Admin Aucun. 200 utilisateurs bannis.
GET /api/v1/admin/users/banned/count Admin Aucun. 200 { "bannedUsers": number }.
GET /api/v1/admin/users/:id Admin id entier positif. 200 profil admin avec moderationLogs, ou 404 USER_NOT_FOUND.
POST /api/v1/admin/users/:id/ban Admin reason requis, 10..1000 caractères. Un admin ne peut pas se bannir lui-même. 200 { "ok": true }.
POST /api/v1/admin/users/:id/unban Admin reason requis, 10..1000 caractères. 200 { "ok": true }.
