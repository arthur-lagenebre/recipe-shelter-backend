Auth

Les sessions community et staff utilisent des cookies, audiences JWT, durées et tables SQL distincts. Le login community pose rs_app_session pour 7 jours par défaut. Le login staff vérifie le mot de passe puis une assertion WebAuthn avec vérification utilisateur avant de poser rs_admin_session pour 8 heures par défaut. Les opérations staff hautement sensibles vérifient en plus que StaffSessions.MfaVerifiedAt date de moins de 5 minutes par défaut ; sinon elles répondent 401 AUTH_RECENT_AUTHENTICATION_REQUIRED. Une nouvelle connexion staff complète renouvelle cette preuve. Les routes sensibles sont limitées par le rate limiter auth.

GET /api/v1/recipes/search Auth optionnelle Mêmes filtres que /api/v1/recipes. Les quatre listes d'identifiants sont séparées par des virgules et dédupliquées. Propositions catalogue

Suggestions de tags et d'ingrédients liées à une recette appartenant au compte community actif. Chaque route est limitée à 10 tentatives par heure et par adresse IP. Une proposition reste en attente de revue et ne crée ni entité canonique, ni association à la recette.

Méthode Route Auth Body / Query Réponse
POST /api/v1/catalog/tag-proposals Community active { "recipeId": number, "name": string }. Nom non vide, 255 caractères maximum. 201 proposition pending.
POST /api/v1/catalog/ingredient-proposals Community active { "recipeId": number, "name": string }. Nom non vide, 255 caractères maximum. 201 proposition pending.

Traitement staff

Toutes les routes ci-dessous exigent une session staff active, catalog.manage et produisent un audit administratif. Chaque décision de revue exige un motif de 10 à 1000 caractères et l'audit partage sa transaction. L'acceptation exige en plus la permission de création du catalogue concerné ; la conversion en alias exige ingredient.alias.manage. Le nom canonique ou l'alias créé provient exclusivement du libellé immuable de la proposition.

Méthode Route Auth Body / Query Réponse
GET /api/v1/admin/catalog-proposals catalog.manage page, limit, status (défaut pending), proposalType, recipeId, authorUserId, q. 200 file paginée.
POST /api/v1/admin/catalog-proposals/tags/:id/accept catalog.manage + tag.create { "groupId": number, "slug"?: string, "description"?: string | null, "reason": string }. 201 proposition accepted liée au tag créé.
POST /api/v1/admin/catalog-proposals/ingredients/:id/accept catalog.manage + ingredient.create { "slug"?: string, "reason": string }. 201 proposition accepted liée à l'ingrédient créé.
POST /api/v1/admin/catalog-proposals/:id/reject catalog.manage { "reason": string }. 200 proposition rejected.
POST /api/v1/admin/catalog-proposals/tags/:id/associate catalog.manage { "targetTagId": number, "reason": string }. 200 proposition merged liée à un tag actif.
POST /api/v1/admin/catalog-proposals/ingredients/:id/associate catalog.manage { "targetIngredientId": number, "reason": string }. 200 proposition merged liée à un ingrédient actif.
POST /api/v1/admin/catalog-proposals/ingredients/:id/alias catalog.manage + ingredient.alias.manage { "targetIngredientId": number, "languageCode": string, "reason": string }. 201 alias créé et proposition merged.

Conflits et propriété

404 CATALOG_PROPOSALS_RECIPE_NOT_FOUND masque aussi une recette appartenant à un autre compte. 409 CATALOG_PROPOSALS_CANONICAL_NAME_EXISTS refuse un nom canonique ou un alias d'ingrédient actif déjà connu. 409 CATALOG_PROPOSALS_ALREADY_PENDING refuse le même nom normalisé déjà en attente pour la recette et le type. Le dépassement de débit renvoie 429 RATE_LIMIT.

Transaction et conservation

La fusion verrouille les relations, remplace uniquement RecipeIngredients.IngredientId, transfère les alias et redirige l'historique. Le DisplayText rédigé par l'auteur et les autres champs de la ligne recette restent inchangés. Le nom français de la source devient un alias protégé de la cible ; un alias normalisé équivalent sur la source ou la cible est réutilisé. La mutation et l'audit avant/après sont validés ensemble, et tout échec annule l'ensemble.

Admin staff management

Cycle de vie logique et rôles des comptes staff. Aucun endpoint ne supprime physiquement un compte et aucun endpoint de réinitialisation MFA n’est exposé dans le back-office. La sortie du staff passe par la désactivation et la révocation de ses accès. Chaque route exige sa permission explicite et produit un audit. Les mutations exigent un motif de 10 à 1000 caractères. La désactivation et le retrait du dernier rôle révoquent globalement les sessions du compte ; tout changement du rôle SuperAdmin exige une authentification forte récente. Un staff ne peut ni se désactiver, ni s’attribuer, ni retirer ses propres rôles. Le dernier SuperAdmin actif ne peut être ni désactivé ni privé de ce rôle par un autre staff.

Méthode Route Auth Body / Query Réponse
GET /api/v1/admin/staff Admin + staff.read Aucun. 200 { "staff": AdminStaffAccount[] }.
GET /api/v1/admin/staff/:staffUserId Admin + staff.read staffUserId entier positif. 200 AdminStaffAccount ou 404 STAFF_USER_NOT_FOUND.
POST /api/v1/admin/staff/:staffUserId/disable Staff récent + staff.disable { "reason": string }, 10..1000 caractères. Cible obligatoirement active. 200 compte désactivé ; toutes ses sessions actives sont révoquées atomiquement. Refus 401 AUTH_RECENT_AUTHENTICATION_REQUIRED sans preuve forte récente, 403 STAFF_DISABLE_SELF_FORBIDDEN sur soi-même. Conflit 409 LAST_ACTIVE_SUPER_ADMIN si un autre staff cible le dernier SuperAdmin actif.
POST /api/v1/admin/staff/:staffUserId/enable Admin + staff.enable { "reason": string }. Cible obligatoirement disabled avec MFA enrôlé. 200 compte réactivé.
POST /api/v1/admin/staff/:staffUserId/roles/:roleCode Admin + staff.role.grant Code de rôle stable et { "reason": string }. 200 compte avec ses rôles mis à jour. Le rôle SuperAdmin exige une preuve forte récente, sinon refus 401 AUTH_RECENT_AUTHENTICATION_REQUIRED. Refus 403 STAFF_ROLE_GRANT_SELF_FORBIDDEN sur soi-même.
DELETE /api/v1/admin/staff/:staffUserId/roles/:roleCode Admin + staff.role.revoke Code de rôle stable et { "reason": string }. 200 compte avec ses rôles mis à jour. Le rôle SuperAdmin exige une preuve forte récente, sinon refus 401 AUTH_RECENT_AUTHENTICATION_REQUIRED. Refus 403 STAFF_ROLE_REVOKE_SELF_FORBIDDEN sur soi-même. Conflit 409 LAST_ACTIVE_SUPER_ADMIN si un autre staff cible le rôle du dernier SuperAdmin actif.
